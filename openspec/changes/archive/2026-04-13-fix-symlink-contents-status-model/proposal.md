# Proposal: Fix Sync-Type-Aware Status For Symlink-Contents Targets

## Intent

AgentSync status currently evaluates every configured destination as if the destination path itself must be a symlink. That model conflicts with `symlink-contents` apply behavior, where AgentSync creates or reuses a real destination directory and manages symlinked child entries inside it.

This mismatch is especially confusing after wizard-driven migrations for commands targets because empty `.agents/commands/` source directories are valid, but `agentsync status` still reports destination directories like `.claude/commands`, `.gemini/commands`, or `.opencode/command` as missing or "not a symlink" problems. The product intent of this change is to make status semantics explicitly sync-type-aware, define the expected status outcome for empty `symlink-contents` sources, and align all user-facing documentation and examples with that model.

## Scope

### In Scope
- Define status semantics per sync type so `symlink`, `symlink-contents`, and existing per-entry sync behaviors are evaluated against their actual apply model instead of one generic directory-symlink rule.
- Clarify how `status` should behave for `symlink-contents` targets when the source directory exists but has zero eligible child entries, including empty commands directories created by `init` or wizard migration.
- Align documentation, examples, and migration guidance so command directory naming and examples consistently use the canonical `.agents/commands/` source while preserving agent-specific destinations such as `.claude/commands`, `.gemini/commands`, and `.opencode/command`.
- Update product examples and status messaging so users can reason about commands targets after migration without contradictory signals.

### Out of Scope
- Redesigning the underlying `apply` sync engine for `symlink-contents` targets.
- Changing the canonical commands directory layout beyond documentation and status-model clarification.
- Introducing new sync types or changing existing destination paths generated by the wizard.

## Approach

Document and implement `status` as a sync-type-aware validator instead of a pure destination-path symlink check.

For `symlink`, status should continue validating the destination path as a single symlink to the expected source.

For `symlink-contents`, status should validate the destination as a managed directory container whose child entries are the managed symlinks. The proposal should explicitly cover the empty-source case: if the source directory exists but has no matching items, the destination directory may legitimately exist as an empty directory and should not be reported as drift solely because it is not itself a symlink.

The change should also normalize product language and examples around commands naming. Documentation and examples should consistently distinguish:
- canonical source directory: `.agents/commands/`
- agent destination directories: `.claude/commands/`, `.gemini/commands/`, `.opencode/command/`

## Affected Areas

| Area | Impact | Description |
|------|--------|-------------|
| `src/commands/status.rs` | Modified | Make status collection and evaluation aware of sync type and directory-container semantics for `symlink-contents`. |
| `src/commands/status_tests.rs` | Modified | Add regression coverage for empty `symlink-contents` sources and non-problematic managed destination directories. |
| `src/linker.rs` | Modified | Reuse or expose source expectation details needed so status can reflect apply semantics accurately. |
| `src/init.rs` | Modified | Align wizard/user-facing summaries for commands targets with the clarified sync-type-aware status model. |
| `README.md` | Modified | Correct command vs commands examples and status expectations in top-level docs. |
| `website/docs/src/content/docs/reference/configuration.mdx` | Modified | Clarify commands semantics, empty-directory behavior, and naming consistency. |
| `website/docs/src/content/docs/reference/cli.mdx` | Modified | Explain what `status` asserts for `symlink` versus `symlink-contents`. |

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Status semantics change could hide real drift for misconfigured `symlink-contents` destinations | Medium | Define explicit rules for expected child entries, stale child symlinks, and invalid non-symlink children in specs/tests before implementation. |
| Documentation updates may miss older singular `command` examples and leave terminology inconsistent | Medium | Audit README, docs site, wizard copy, and examples together as one acceptance set. |
| JSON or textual status output changes may affect user expectations or tests | Medium | Treat output behavior as an intentional product change, add regression tests, and document new semantics clearly. |

## Rollback Plan

If the new status model causes regressions or unclear output, revert the status-specific logic and related copy changes to restore the previous destination-path validation behavior. Keep the rollback scoped to status and documentation so apply behavior remains untouched.

## Dependencies

- Existing OpenSpec requirements in `openspec/specs/core-sync-engine/spec.md` and `openspec/specs/documentation/spec.md` will need follow-up deltas to codify the new status and documentation expectations.
- Issue context from GitHub issue `#342` should remain the product reference for the migration confusion this proposal addresses.

## Success Criteria

- [ ] `agentsync status` evaluates configured targets according to their sync type rather than assuming every destination path must itself be a symlink.
- [ ] A `symlink-contents` commands target with an existing empty source directory does not report the destination directory as missing or "not a symlink" solely because the directory contains no managed entries yet.
- [ ] Status output and tests clearly distinguish valid empty `symlink-contents` containers from real drift conditions.
- [ ] README, docs, and examples consistently use the canonical `.agents/commands/` naming and accurately explain agent-specific destination paths, including `.opencode/command`.
- [ ] Wizard and migration guidance no longer leave users with contradictory mental models for commands targets after `apply` and `status`.
